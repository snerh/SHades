4. Измерение как @async-задача

function measure_spectrum_async(
    lockin,
    mono,
    params::ScanParams,
    ch::EventChannel,
    ctrl::MeasurementControl
)
    try
        wl = params.wl_start:params.step:params.wl_stop
        signal = Float64[]

        for (i, λ) in enumerate(wl)
            ctrl.stop && begin
                put!(ch, MeasurementStopped())
                return
            end

            set_wavelength!(mono, λ)
            sleep(params.integration_time)

            val = read_signal(lockin)
            push!(signal, val)

            spec = Spectrum(collect(wl[1:i]), copy(signal))
            put!(ch, StepResult(i, spec))
        end

        put!(ch, MeasurementFinished())

    finally
        close(ch)
    end
end

    только put!

    никакого Gtk

    канал всегда закрывается

1. Где реально возникают race condition’ы

В твоей архитектуре их всего три класса. Остальное — мифы.
1. Флаг остановки (ctrl.stop)

Проблема:
UI пишет ctrl.stop = true, async-задача читает ctrl.stop.

Риск:
Минимальный. В Julia чтение/запись Bool атомарны.

Когда становится проблемой:

    несколько async-задач,

    или сложные структуры.

Вывод:
Для одного измерения — безопасно.
2. Данные спектра (массивы)

Это главный реальный race.

❌ Плохо:

put!(ch, Spectrum(wl[1:i], signal))

UI и измерение будут читать/писать один и тот же массив.

Симптомы:

    график «дрожит»,

    длины не совпадают,

    случайные ошибки.

✅ Правильно:

spec = Spectrum(copy(wl[1:i]), copy(signal))
put!(ch, spec)

Правило:
👉 В канал — только иммутабельные или скопированные данные.
3. Закрытие канала

Проблема:
Кто закрывает Channel и когда.

❌ Плохо:

    закрывать канал из UI,

    закрывать в середине цикла.

✅ Правильно:

    канал закрывает только производитель,

    всегда в finally.

finally
    close(ch)
end

UI никогда не вызывает close.
2. Типичные ошибки и как их чинить
Ошибка 1. take! в UI + stop

UI зависает навсегда, если:

    stop = true,

    но канал не закрыт.

Решение:
Всегда закрывать канал при выходе измерения.
Ошибка 2. UI обновляет Gtk из async-задачи

Gtk не thread-safe.

❌ Плохо:

@async update_plot(spec)

✅ Правильно:

    async читает канал,

    UI-обновление выполняется в основном loop’е Gtk
    (через GLib.idle_add, если используешь Gtk).

Ошибка 3. Несколько потребителей одного Channel

Channel — очередь, не broadcaster.

Если ты читаешь канал:

    в UI,

    и в логгер,

один из них «съест» событие.

Решения:

    Один consumer → он дальше раскидывает.

    Или несколько каналов.

3. Stop vs Cancel vs Kill
❌ Никогда не делай

Base.throwto(task, InterruptException())

Ты:

    оставишь прибор в неопределённом состоянии,

    не закроешь канал,

    сломаешь UI.

✅ Правильный cancel

ctrl.stop = true

И проверка в измерении:

ctrl.stop && return

4. Где проверять stop (важно)

Проверяй до и после блокирующих операций:

for ...
    ctrl.stop && break

    set_wavelength!(...)
    sleep(...)

    ctrl.stop && break

    val = read_signal(...)
end

Иначе stop будет «запаздывать».
5. Backpressure и размер Channel

Channel{MeasurementEvent}(10)

Что это даёт:

    UI не успевает → измерение притормаживает.

    Нет утечки памяти.

❌ Channel(Inf) — почти всегда ошибка.
6. Потокобезопасный минимум (чеклист)

Если всё ниже — true, у тебя нет race condition’ов:

В канал передаются копии данных

Канал закрывает только измерение

UI не пишет в канал

Gtk вызывается только из UI

Stop — через флаг, не через exception

    Один producer → один channel

7. Архитектурный маркер зрелости

Хороший признак:

    ты можешь заменить UI на @async for ev in ch println(ev) end

    и измерение будет работать так же.

Плохой:

    измерение знает, что его рисуют.

8. Что логично разобрать дальше

Следующие разумные шаги:

    Pause без busy-wait

    Таймауты приборов

    Гарантированный reset приборов при stop

    Тест async-измерений без GUI

Скажи, что из этого идём разбирать.
